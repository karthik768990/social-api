# Social API — Redis Guardrails Microservice

A Spring Boot 3.x microservice acting as a central API gateway with a Redis-backed
virality engine, atomic concurrency guardrails, and a smart notification batching system.

---

## Tech Stack

- **Java 17** + **Spring Boot 3.2**
- **PostgreSQL 16** — source of truth for all content
- **Redis 7** — gatekeeper for all counters, cooldowns, and notifications
- **Spring Data Redis** — Lua script execution via `RedisTemplate`
- **Docker Compose** — one-command local setup

---

## Quick Start

```bash
# 1. Start PostgreSQL and Redis
docker-compose up -d

# 2. Build and run the app
./mvnw spring-boot:run

# 3. Import postman_collection.json into Postman and start testing
```

The app runs on `http://localhost:8080`.

## My Approach

The service is split into four clear layers matching the assignment phases:

**PostgreSQL** acts as the permanent source of truth — it stores all users, bots,
posts, and comments. **Redis** acts as the gatekeeper — every guardrail check,
counter, cooldown, and notification queue lives exclusively in Redis. The Spring Boot
application itself is fully stateless with zero in-memory counters or static variables,
meaning it can be scaled horizontally without breaking any guardrail.

When a bot tries to comment, the request passes through three Redis checks in order
before a single DB write is attempted. Only after all three pass does the comment get
written to PostgreSQL. If the DB write fails after Redis has already incremented the
bot counter, a compensating DECR is issued to roll Redis back, keeping both stores
consistent.

The virality score is updated in Redis immediately after every interaction using
atomic INCRBY, and is returned live alongside every post response.

The notification engine uses a Redis TTL key per user to decide whether to send
immediately or batch. A Spring @Scheduled CRON task sweeps every 5 minutes,
pops all pending messages atomically, and logs a summarized notification.

## How Thread Safety Is Guaranteed for the Atomic Locks (Phase 2)

This is the most critical design decision in the entire service. The problem is
straightforward: under 200 concurrent bot requests hitting the same post at the
same millisecond, a naive check-then-act pattern will fail. Two threads can both
read the counter as 99, both decide it is under the cap, and both proceed —
resulting in 101 comments instead of 100.

### Why not Java locks?

Using `synchronized` blocks or `ReentrantLock` would work on a single JVM instance,
but it directly violates the statelessness requirement. The moment you run two
instances of the app, each instance has its own lock and they cannot coordinate.
The counter would overflow on any multi-instance deployment.

### Why not Redis MULTI/EXEC transactions?

Redis transactions using WATCH + MULTI/EXEC work with optimistic locking. If two
threads try to modify the same key at the same time, one of them detects a conflict
and retries. Under 200 simultaneous requests, this causes a thundering-herd
problem — every request keeps retrying, latency spikes, and some requests may
never complete. It is not suitable for high-concurrency guardrails.

### The solution: Lua scripts via Redis EVAL

All three atomic locks use Lua scripts executed directly on the Redis server via
the EVAL command. Redis is single-threaded internally, which means it processes
one Lua script at a time with no interruption. The script itself performs the read,
the comparison, and the write as a single indivisible operation. No other command
from any other thread can be interleaved between those steps.

**Horizontal Cap (max 100 bot replies per post):**

The Lua script reads the current bot count, compares it against 100, and either
returns 0 (rejected) or increments and returns the new value (allowed) — all in
one atomic round-trip. Under 200 concurrent threads, Redis serializes all 200
script executions. The 101st execution will always read exactly 100 and be rejected.
It is mathematically impossible for the counter to exceed 100.

```lua
local current = tonumber(redis.call('GET', KEYS[1]) or '0')
if current >= tonumber(ARGV[1]) then
  return 0
end
return redis.call('INCR', KEYS[1])
```

**Cooldown Cap (bot cannot interact with same human within 10 minutes):**

The Lua script checks if the cooldown key exists and, if not, sets it with a 10-minute
TTL in the same atomic operation. This eliminates the race condition where two
concurrent requests both see "key does not exist" and both set the cooldown,
both believing they are the first interaction.

```lua
if redis.call('EXISTS', KEYS[1]) == 1 then
  return 0
end
redis.call('SET', KEYS[1], '1', 'EX', ARGV[1])
return 1
```

**Vertical Cap (max thread depth 20):**

This cap does not require Redis at all. The depth level is provided in the request
payload and is checked in Java before any Redis or DB call is made. It is a pure
arithmetic check with no concurrency concern.

### Summary

| Approach | Why it was rejected |
|---|---|
| `synchronized` / Java locks | Breaks statelessness, fails on multi-instance deployment |
| `HashMap` / static variables | Resets on restart, not shared across instances |
| Redis `WATCH` + `MULTI/EXEC` | Thundering herd under high concurrency, retry storms |
| **Lua scripts via `EVAL`** | Atomic on Redis server, zero retries, works across all instances |

## Phase 3: Notification Engine

When a bot interacts with a user's post, the service checks if a 15-minute
notification cooldown key exists for that user in Redis.

- **No active cooldown:** Logs the push notification immediately to console and
  sets the 15-minute cooldown key.
- **Active cooldown:** Pushes the notification string into a Redis List
  (`user:{id}:pending_notifs`) for batching.
